## 二、原因分析

### 2.1 撤单原因分布

**Polymarket（665 次撤单）：**

| 原因 | 次数 | 占比 | 说明 |
|------|------|------|------|
| rewards-bid-level-2（depth level 2） | 416 | **62.6%** | 正常替换，挂在第 2 档 |
| rewards-bid-level-3 | 86 | 12.9% | 挂在第 3 档 |
| rewards-bid-level-4~11 | 163 | 24.5% | 其他深度档位 |

**Predict（1439 次撤单）：**

| 原因 | 次数 | 占比 | 说明 |
|------|------|------|------|
| predict-points-bid-level-3 | 1381 | **95.9%** | 正常替换，挂第 3 档 |
| predict-points-bid-level-4 | 58 | 4.1% | 正常替换，挂第 4 档 |

| 指标 | Polymarket | Predict |

### 2.4 核心发现

**Polymarket 运行非常健康：**
- 96.4% 撤单率是 cash 模式的正常现象——每轮周期替换订单以维持最优深度
- 0 次周期慢警告，无新单暂停，A-2/A-3 保护在正常工作
- skip-existing ~2000/小时说明快速重挂模式运行高效
- 仅 1 次 WS 盘口即时撤单说明市场质量好，不需要频繁撤退
- 15 次拒单（2.2%）全部是最终盘口检查不通过，正常

**Predict 有系统性性能问题：**
- **裸奔防守撤单 898 次**——大量订单在盘口数据缺失/过期时被撤
- **REST 验证失败 445 次**——Predict API 连接不稳定或被限流
- **盘口不可用 460 个 token**——无法获取订单簿数据
- **51 次周期慢警告 + 609 次新单暂停**——周期频繁超过 45 秒
- 周期耗时极不稳定（16-113 秒），峰值时期几乎无法正常工作
- 15:00-17:00 峰值 391 次提交/小时，之后引擎可能因故障自动停止

## 三、结论与建议

### 3.1 Polymarket — 无瓶颈，可扩容

当前 6 市场 $39/单的配置非常健康。三层过滤运行良好，A-2/A-3 保护触发频率低说明市场质量筛选有效。

**建议：maxMarkets → 10，maxDailyLossUsd → 12**
- 每周期路由扫描 60 个候选（不变），新增 4 个订单提交约 +2 秒
- 日亏损上限 $12 可承受 2.5 次被吃（每次最多亏 $4.68）
- 不需要改任何代码，只改配置

### 3.2 Predict — 需修复系统性性能问题

预测市场的核心问题不是保护层不足，而是 **API 可靠性太低**导致大量裸奔撤单。

**根因链条：**
```
Predict API 响应慢/失败
  → 订单簿频繁不可用（460 次）
  → 裸奔防守撤单（898 次）+ REST 验证失败（445 次）
  → 周期拉长到 45-113 秒
  → 触发新单暂停（609 次）
  → 有效挂单时间大幅缩短
  → 收益下降 + 被吃风险增加
```

**紧急建议：**
1. Predict `staleBookMs` 从 2000 → 6000（减少因 WS 缓存过期导致的 REST 回退）
2. 启用 `cashSupportWindowCents: 1`（补上后方支撑，和 Polymarket 对齐）
3. 检查 Predict API 是否有速率限制，需要加请求间隔或重试退避
4. 考虑降低 `maxMarkets: 15 → 10` 以减少周期负载

### 3.3 撒单率本身不是问题

两边的撤单率（96%/99%）看起来吓人，但这是 **cash 模式的正常行为**——每轮周期策略都会计算最优挂单价格，旧单被替换为新单。Polymarket 用快速重挂（skip-existing）避免无谓的撤单-重挂循环，Predict 目前没有这个机制所以撤得更狠。
